# Plan 19-03: proxy.ts → Supabase Auth admin ロール移行（Basic Auth 廃止）

**Phase:** 19 — Supabase Auth + メンバー自己管理
**Plan:** 19-03
**Wave:** 1
**Status:** draft

## Objective

`src/proxy.ts` を書き換え、`/admin` の認証を Basic Auth から Supabase Auth の
`app_metadata.role === 'admin'` チェックに完全移行する。
あわせて `/my` ルートのログイン必須ガードも proxy.ts に追加する。
`ADMIN_PASSWORD` 環境変数は不要になる。

## Requirements Addressed

- AUTH-04: /adminへのアクセスがSupabaseユーザーロール（admin role）で制御され、Basic Authが不要になる

## Pre-implementation Notes

- Next.js 16 では `proxy.ts` が Edge Middleware のエントリポイント（`middleware.ts` は削除済み）
- Edge runtime で `@supabase/ssr` の `createServerClient` を使い `getUser()` でセッション確認
- `getUser()` はサーバーサイドでトークンを検証するため安全（`getSession()` は使用禁止）
- admin ロールは Supabase の `user.app_metadata.role === 'admin'` で確認する（D-01）
- admin ユーザーの設定は人間作業（Supabase Dashboard または service_role API で app_metadata をセット）
- matcher に `/my` と `/my/:path*` を追加（D-07）
- Basic Auth ロジック（`ADMIN_PASSWORD` 参照）を完全に削除する（D-01）

## Prerequisite: admin ユーザーのロール設定

**Phase 19 デプロイ後にユーザーが実施する必要がある。**

Supabase Dashboard > Authentication > Users で管理者ユーザーを選択し、
App Metadata に以下を設定する:
```json
{ "role": "admin" }
```

またはサービスロールを使った curl:
```bash
curl -X PUT 'https://<project>.supabase.co/auth/v1/admin/users/<user-id>' \
  -H 'apikey: <service_role_key>' \
  -H 'Authorization: Bearer <service_role_key>' \
  -H 'Content-Type: application/json' \
  -d '{"app_metadata": {"role": "admin"}}'
```

## Success Criteria

1. `src/proxy.ts` が Supabase Auth を使い、`/admin` へのアクセスで `app_metadata.role !== 'admin'` の場合 `/login` にリダイレクトする
2. `src/proxy.ts` が `/my` へのアクセスで未ログインの場合 `/login` にリダイレクトする
3. `src/proxy.ts` に `ADMIN_PASSWORD` の参照が存在しない
4. `config.matcher` に `/my` と `/my/:path*` が含まれる
5. `npm run build` がエラーなく完了する

## Tasks

### Task 1: `src/proxy.ts` を Supabase Auth ガードに書き換える

**File:** `src/proxy.ts`（全書き換え）

```typescript
import { createServerClient } from '@supabase/ssr'
import { NextRequest, NextResponse } from 'next/server'

export async function proxy(request: NextRequest) {
  let response = NextResponse.next({ request })

  const supabase = createServerClient(
    process.env.NEXT_PUBLIC_SUPABASE_URL!,
    process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY!,
    {
      cookies: {
        getAll() {
          return request.cookies.getAll()
        },
        setAll(cookiesToSet) {
          cookiesToSet.forEach(({ name, value }) => request.cookies.set(name, value))
          response = NextResponse.next({ request })
          cookiesToSet.forEach(({ name, value, options }) =>
            response.cookies.set(name, value, options)
          )
        },
      },
    }
  )

  // セッションをリフレッシュ（getUser() はサーバー側検証でセキュア）
  const {
    data: { user },
  } = await supabase.auth.getUser()

  const { pathname } = request.nextUrl

  // /admin: admin ロール必須
  if (pathname.startsWith('/admin')) {
    if (!user || user.app_metadata?.role !== 'admin') {
      return NextResponse.redirect(new URL('/login', request.url))
    }
  }

  // /my: ログイン必須
  if (pathname.startsWith('/my')) {
    if (!user) {
      return NextResponse.redirect(new URL('/login', request.url))
    }
  }

  return response
}

export const config = {
  matcher: ['/admin', '/admin/:path*', '/my', '/my/:path*'],
}
```

## Notes

- `getUser()` はネットワーク呼び出しを伴うため全リクエストに若干のレイテンシが追加される（50ms程度）
  - 許容範囲: /admin と /my は管理・ユーザー向けページで低頻度アクセスのため問題なし
- Basic Auth のブラウザ認証ダイアログが消えるため、既存の管理者は Magic Link でログインし直す必要がある
- `ADMIN_PASSWORD` 環境変数は削除可能だが、Phase 21 のクリーンアップで実施する（Vercel 環境変数の削除は手動）
- session リフレッシュ（setAll）がないとトークンの有効期限切れ時に認証が途切れる。setAll でレスポンスのCookieを更新するパターンを忘れずに実装すること
