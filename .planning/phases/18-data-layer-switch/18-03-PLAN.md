# Plan 18-03: middleware.ts → proxy.ts リネーム

**Phase:** 18 — データレイヤー差し替え + 長期記事履歴
**Plan:** 18-03
**Wave:** 1
**Status:** draft

## Objective

`src/middleware.ts`（Basic Auth ロジック）を `src/proxy.ts` にリネームし、
`src/middleware.ts` は `proxy.ts` から re-export する薄いラッパーとして再作成する。

Next.js は `src/middleware.ts` を特別なファイルとして認識するため、
このリネームにより「ロジックは proxy.ts に分離、エントリポイントは middleware.ts」という構造になる。
Phase 19 で Supabase Auth ロールによる管理画面制御に移行する際、`middleware.ts` の書き換えだけで完結する（D-06）。

## Requirements Addressed

- D-06: src/middleware.ts を src/proxy.ts にリネームする（Next.js 16対応、Phase 19 準備）

## Pre-implementation Notes

- `src/middleware.ts` は Next.js が自動的に Edge Middleware として認識する予約ファイル名
- ファイルを単純に削除・リネームするだけでは Next.js がミドルウェアを読み込めなくなり /admin 認証が無効化される
- 対策: `src/proxy.ts` にロジックを移動し、`src/middleware.ts` は re-export のみにする
- Basic Auth（ADMIN_PASSWORD）は Phase 19 まで継続して機能させる必要がある

## Success Criteria

1. `src/proxy.ts` が作成され、Basic Auth ロジック（`middleware` 関数・`config` オブジェクト）を含む
2. `src/middleware.ts` が `proxy.ts` から `middleware` と `config` を re-export している
3. `/admin` へのアクセスが引き続き Basic Auth で保護されている（動作変更なし）
4. `npm run build` がエラーなく完了する

## Tasks

### Task 1: `src/proxy.ts` を作成する

**File:** `src/proxy.ts`（新規作成）

`src/middleware.ts` の現在の内容をそのままコピーする:

```typescript
import { NextRequest, NextResponse } from 'next/server'

// D-01: ADMIN_PASSWORD のみで認証（ユーザー名不要）
// D-02: Next.js Edge Middleware で /admin をインターセプト
// D-03: 認証失敗時は 401 + WWW-Authenticate ヘッダー
export function middleware(request: NextRequest) {
  const authHeader = request.headers.get('authorization')
  const adminPassword = process.env.ADMIN_PASSWORD ?? ''
  const encoded = btoa(`:${adminPassword}`)

  if (authHeader !== `Basic ${encoded}`) {
    return new NextResponse('Unauthorized', {
      status: 401,
      headers: { 'WWW-Authenticate': 'Basic realm="Admin"' },
    })
  }

  return NextResponse.next()
}

export const config = {
  matcher: ['/admin', '/admin/:path*'],
}
```
