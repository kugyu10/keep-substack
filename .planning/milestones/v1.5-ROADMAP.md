# Milestone v1.5: Member Auth + Supabase Migration

**Status:** SHIPPED 2026-05-17
**Phases:** 17-21
**Total Plans:** 11

## Overview

Supabase完全移行・ログイン・メンバー自己管理を実現し、管理者依存を排除してメンバーが自律的に参加できるサービスにする。Upstash Redis（KV）を廃止してPostgreSQLに完全移行し、Magic Linkログイン・/myページ・管理者ロール制御を実装した。

## Key Accomplishments

1. Supabase PostgreSQL（4テーブル + RLS）+ 3種クライアントライブラリ基盤を構築
2. getMembers()/saveArticles()等の既存シグネチャを維持しつつSupabaseに完全移行
3. Magic Linkログイン（/login-51cf21389c56/ + /auth/callback）実装
4. /myページでメンバー自己管理（substackId紐付け・プロフィール編集）を実現
5. /adminをBasic Auth → Supabase authロール（app_metadata.role=admin）制御に移行
6. @upstash/redis完全削除でコードベースをSupabase一本化

## Phases

### Phase 17: Supabaseスキーマ + RLS設定 + KVデータ移行

**Goal**: Supabase PostgreSQLのテーブル定義・RLSポリシー・クライアントライブラリが揃いアプリがSupabaseに接続できる状態になり、既存Upstash RedisのデータがPostgreSQLに過不足なく移行されている
**Depends on**: Phase 16 (v1.4完了)
**Plans**: 3プラン

Plans:

- [x] 17-01: PostgreSQL DDL + RLS SQL スクリプト（supabase/schema.sql）
- [x] 17-02: Supabaseクライアントライブラリ基盤（server.ts / client.ts / admin.ts）
- [x] 17-03: KVデータ移行スクリプト（scripts/export-to-sql.ts）

**Key Decisions:**
- service_roleはBYPASSRLS権限を持つためINSERT/UPDATE/DELETE用ポリシーは不要
- articlesテーブルのpublication_idはmembers.publication_idへのFK（ON DELETE CASCADE）
- link列のUNIQUE制約で記事の重複排除（Phase 18のCronでON CONFLICT DO NOTHINGと組み合わせ）

### Phase 18: データレイヤー差し替え + 長期記事履歴

**Goal**: アプリの読み書きが完全にSupabaseを参照し、Cronが長期記事を累積保存できる状態になる
**Depends on**: Phase 17
**Plans**: 3プラン

Plans:

- [x] 18-01: kvMembers.ts — Supabase差し替え（シグネチャ維持）
- [x] 18-02: kvArticles.ts — Supabase差し替え + articlesスキーマ拡張（image_url）
- [x] 18-03: middleware.ts → proxy.ts リネーム（Next.js 16対応）

**Key Decisions:**

### Phase 19: Supabase Auth + メンバー自己管理

**Goal**: メンバーがMagic Linkでログインして自分のプロフィールを自己管理でき、管理者への依頼なしに参加登録できる
**Depends on**: Phase 18

### Phase 20: 管理画面チームチェックボックス
